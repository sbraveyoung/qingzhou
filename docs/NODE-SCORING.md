# 节点多维打分择优 — 设计（v1，2026-07-07 拍板）

> 结论先行：**可行、有价值，但价值不在「维度多」，在三件事**——
> ① 稳定性维度避开「延迟低但抖动大」的节点（纯延迟择优最常见的翻车点）；
> ② 用你真实访问的域名当测量目标，分数反映真实体验而非 generate_204；
> ③ 倍率从「延迟相近才看」升级为连续成本项，流量成本可量化 → 直接支撑竞品对比宣发。

## 0. 现状底账（已核实代码）

现有算法 = **延迟单主维 + 倍率 tiebreaker + 地区硬约束 + 黏性滞后**：
- 全量测直连 TCP 握手 RTT → 「最快 ±30ms」视为相当（band），band 内选低倍率，同倍率再比延迟；
- VPN 运行中另有经代理精选（tieBand 200ms）；
- 换当前节点必须「快 ≥50ms 且降幅 ≥30%」（`autoSwitchWorthRestart`）。

可直接复用的按节点数据：直连延迟、经代理延迟、倍率（`rateForComparison`，未标注按 1.0）、
地区（名字/旗帜 emoji 解析）、被动带宽峰值（`observedPeakDownBps`，当前节点跑流量时顺带记录）。
**空白**：每节点只存最近一次测量（无历史序列 → 没法算稳定性）；无按节点的丢包/失败率留存。

## 1. 打分模型

`score(node) = Σ w_d × s_d(node)`，每维 `s_d ∈ [0,100]`，**锚点归一**（不随节点池相对漂移，可解释）。

| 维度 | 权重(均衡档) | 数据源 | 归一锚点 |
|---|---|---|---|
| 延迟 L | 0.45 | 经代理 + 直连**按分数混合**（2026-07-07 增强）：两者都有且经代理新鲜（≤24h）→ `0.7×score(经代理) + 0.3×score(直连×1.3)`，各自先过锚点归一再混合（不混原始 ms，量纲锚点不同）；只有经代理 → 全权重；只有直连 → ×1.3 惩罚；经代理 `lastProxiedTestedAt` 距今 >24h → 不参与（陈旧的经代理数字比新鲜直连更误导） | ≤50ms→100 · 100→85 · 200→60 · 400→30 · ≥800→0（分段线性） |
| 稳定性 S | 0.30 | 每节点测量历史（环形 20 条）+ **burst 丢包率**（2026-07-07 增强）：直连测速每节点串行 3 次独立 TCP 握手，延迟取成功样本中位数（偶数取较小侧）、丢包率 = 失败/3 | `100 × (1 − 平均丢包率) × (1 − 0.5×变异系数)`；无 lossFraction 的老样本按「成功=0 / 整轮失败=1」折算；样本<3 → 70（中性） |
| 带宽 B | 0.15 | `observedPeakDownBps` 被动峰值，7 天半衰期衰减 | ≥8MB/s→100 · 2MB/s→60 · 0.5MB/s→30 · 无数据→60（中性，不惩罚没轮到的节点） |
| 成本 C | 0.10 | `rateForComparison` | ≤0.5x→100 · 1x→80 · 2x→40 · ≥5x→10 |

**地区**不进打分：`preferredRegion`/`excludedRegions` 保持**硬约束**（先过滤后打分）。
「域名画像 → 地区亲和」维度留到 P3——「哪个地区访问某服务最好」需要数据积累，v1 别拍脑袋。

**域名画像的正确用法**（你想法的核心，我建议这样落）：不是给维度加权，而是给**测量目标**加权——
经代理延迟的探测目标从固定 Cloudflare 改为**你的 Top-3 代理域名**（从 30 天 DomainDailyHistory
按连接数取走 proxy 的域名），`L = Σ(域名权重 × 该域名延迟) / Σ权重`。
延迟维度就天然是「你常用的东西快不快」；带宽被动观测本来就是真实流量，同理免费获得画像属性。

**切换决策（黏性升级，不推倒现有哲学）**：
- 分数滞后：`newBest.score − current.score ≥ 8` **且连续 2 轮成立**才切（幅度+持续双滞后）；
- 当前节点连续 2 次测量失败 → 无条件立即切（可用性 > 稳定）；
- 全量重启路径仍叠加现有 `autoSwitchWorthRestart` 闸；无感热切换路径只用分数滞后。

**UI 原则**：不暴露权重滑杆，只给三档预设——「速度优先 / 均衡 / 省流量优先」
（映射三组权重，省流量档 C 提到 0.30）；节点详情显示分数构成条 + 择优日志写「为什么选它」
（透明度是信任来源，也是宣发差异点）。

## 2. 新增数据结构（唯一的地基工程）

```
NodeMetricsHistory：每节点环形 100 条 {at, latencyMs?, proxiedMs?, lossFraction?}
（P2：20→100，为回放对比实验攒足够长的测量序列；老文件 ≤20 条照常解码、累积到 100 才挤）
lossFraction = 本轮 burst 丢包率（失败/3）；老文件 JSON 无此字段照常解码（Codable 可选）
落盘 node-metrics.json（本地瞬态，不上 iCloud，同 domain-history 待遇）
体量：500 节点 × 100 条 × ~48B ≈ 2.4MB；全在主 App，不碰扩展 50MB 红线
```

**burst 只做直连**：经代理测速不 burst —— 每次都真实走节点发 HTTP，3 倍成本不值当；
经代理只贡献延迟维，不贡献丢包。整轮时长上界 3×5s×节点数/8（并发窗口语义不变），可接受。

## 3. 分阶段（每阶段独立可验收）

- **P1 地基**：NodeMetricsHistory 记录管道 + 稳定性维度 + 打分框架替换
  `bestPreferringLowRate` 内核（外层地区约束/滞后/触发时机全不动），默认均衡档。
- **P2 画像**：Top-3 域名探测目标 + 带宽维度接入 + 三档预设 UI + 「为什么选它」。
- **P3 远期**：地区亲和维度；端侧 AI 动态调权（接你之前的端侧 AI 想法）。

## 4. 竞品对比怎么做（#5）

两条路，先易后难：

**A. 离线回放仿真（P1 数据管道的免费副产品）**：NodeMetricsHistory 积累 ≥1 周后，
用同一份测量序列回放两个策略——「仅直连 ping 最低」（竞品基线）vs 我们的打分——每轮各自选节点，统计：
- 用户加权延迟（对 Top 域名）→ **「提速 X%」**
- 所选节点倍率 × 实际字节 → **「同样用量流量费省 Y%」**
- 切换次数 → **「无谓切换少 Z 次」**

**B. 在线 A/B（自己设备，可选加强）**：诊断页加实验开关，每小时交替策略、对 Top 域名发探测并记录
字节×倍率，7 天自动出报告。

**诚实边界（宣发要守住）**：数字依赖机场质量与个人使用画像——文案写「实测样例：在 N 节点
混合倍率订阅下…」，不做普适承诺；不点竞品名，说「与『仅按 Ping 择优』策略相比」。

## 5. 风险与取舍

- **不做主动吞吐测试**：下载测速烧的是倍率流量（2x 节点测 50MB = 100MB 配额），被动观测够用；
- 探测频率不变（沿用现有 30 分钟档），只是目标更换 → 流量成本几乎不增；
- 打分再好也治不了机场本身抽风——稳定性维度只能「事后避开」，冷启动仍靠延迟。
