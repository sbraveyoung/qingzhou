# 经代理延迟：为什么你的测速数字可能在骗你

> 轻舟技术札记 · 2026-07
> 太长不看：**直连延迟 = 节点离你多远；经代理延迟 = 你用它上网到底多快。**
> 前者测不出出口绕路，更测不出密码错误的"假活节点"。轻舟两个都给你，
> 并默认用后者做自动择优的最终裁决。

## 常见的测速，测的其实不是"上网速度"

市面上绝大多数客户端的"延迟测试"，做的是这么一件事：

```
你的设备 ──── TCP 握手 ────▶ 节点服务器
              ⏱ 32 ms
```

设备直接对节点的 IP/端口发一次 TCP 握手，量出往返时间。这个数字**只回答一个问题**：
"节点服务器离我近不近、这台机器活着没"。

但你真正关心的链路是这样的：

```
你的设备 ──协议握手──▶ 节点服务器 ──节点的出口线路──▶ 目标网站
         （TLS/trojan/vmess…）      （可能绕地球半圈）
```

于是直连延迟有两个致命盲区：

**盲区一：出口绕路。** 节点在香港、离你 30ms，但它的出口线路绕美国——你访问东京的服务，
流量走了 香港→洛杉矶→东京。直连延迟 30ms 光鲜亮丽，实际打开网页比隔壁 80ms 的直连节点慢一倍。

**盲区二：假活节点。** 订阅里密码已过期 / 配置早失效的节点，它的服务器还开着、端口还监听着
——TCP 握手照样成功，直连延迟照样漂亮。trojan 这类协议的服务端出于抗探测设计，
**不会回"认证失败"**，只是把你的流量静默丢弃。你切过去：显示已连接，然后……全网没有响应。
对着一排绿色的延迟数字，你根本猜不到哪个是坑。

## 轻舟的做法：真实走一遍

「经代理延迟」在 VPN 运行时可用：轻舟让隧道扩展里的 xray 为目标节点起一个**临时实例**，
携带完整协议（TLS 握手、trojan/vmess/vless/ss/hysteria2 认证）**真实通过这个节点**
访问一次轻量探测端点（默认 Cloudflare 的 `generate_204`，可在设置里改），量出全链路耗时：

```
你的设备 ═══ 完整协议 ═══▶ 节点 ═══ 出口线路 ═══▶ google.com
                    ⏱ 全链路 412 ms
```

这个数字包含协议握手、节点转发、出口线路质量的全部真相。它天然比直连大（多了协议开销），
所以不要拿两者比大小——要**分别看、对照看**：

| 直连 | 经代理 | 结论 |
|---|---|---|
| 30 ms | 350 ms | 正常，好节点（全链路含协议开销，数百 ms 属正常区间） |
| 30 ms | 1800 ms | 出口绕路 / 节点过载——换 |
| 30 ms | 测试失败 | **假活节点**（凭据错误 / 已下线）——直连数字再好看也别用 |
| 200 ms | 500 ms | 节点远，但出口干净——跨国场景常见，能用 |

在轻舟里，节点列表右侧上下两枚延迟徽章：上面是直连（无图标），下面带 ⋱ 图标的就是经代理。
长按节点可单测，节点页「⋯」菜单可批量测。

## 自动择优：用体感数字做最终裁决

光展示还不够。轻舟的自动择优默认分两步：

1. **海选**：全量节点并发直连测速（快，几秒出结果）；
2. **决赛**：VPN 运行中，**所有直连结果为绿色的节点**逐个真实走一遍代理，
   用经代理延迟选出真冠军——**测不通的候选当场淘汰**。假活节点从此永远进不了
   你的"当前节点"。（绿色 = 到节点的线路健康，值得花时间实测出口；
   黄/红直连本身已说明线路差，不浪费实测名额。）

为什么决赛是串行、不并发？因为它跑在 iOS 的 Network Extension 里——苹果给这个进程
只有 **50MB 内存**的死限额，超了直接杀进程（= 你的 VPN 断流）。每个临时 xray 实例
都吃内存，所以我们一个测完再测下一个、测完即焚——这是在"选得准"和"扩展别爆"
之间踩出来的平衡点。

## 连接了却没网？哨兵会告诉你

同样的思路也用在连接本身：节点配置合法 ≠ 节点可用。轻舟在每次连接 / 切换成功后，
自动经隧道探测一次连通性——如果"已连接"但探测两次都失败，你会收到明确的警告
（"节点可能已失效或凭据错误"），而不是对着绿色开关排查半天路由器。
